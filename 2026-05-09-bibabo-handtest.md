# 2026-05-09 后端手测协作中提取的工作习惯

这次任务:按 `bibabo-后端手测指南.md` 对 `https://api.bibabo.ai` 跑 curl 测试,跑完发现 3 个 issue,分类 file 完之后写报告。下面是这一轮观察到的 chichu 的偏好,留给以后跟他打交道的 AI 参考。

## 关于他自己

- vibe coder 路线,工程基础薄弱。能跑命令、看 JSON、看 HTTP 状态码,但不深入懂工具内部原理
- 讲技术细节要用类比 + 具体例子 + 可直接复制的代码片段。别堆术语
- 看到陌生概念会直接问(比如"console 是什么"),不装懂 — 你也别假设他懂

## 环境

- Windows + Git Bash 是唯一终端,不用 PowerShell 不用 cmd
- macOS 专属命令别给(`brew`、`pbcopy`、`sed -i ''`、`date -v`),要给 Win 等价物
- 文档里出现 `python3` 在 Win 上要换成 `python`(标准 Python.org Win 装好后没 `python3` 这个名字)
- 重启 Git Bash 会丢环境变量,但他评估后选择不持久化 — 接受这个轻微 friction,不要为此过度工程化

## 沟通

- 用人话写。不堆术语,不过度结构化
- UI 元素优先用中文按钮名 — 浏览器开发者工具那个标签叫"控制台"不叫 Console
- 表格在很多场合(钉钉、微信、邮件、富文本编辑器)复制粘贴会坏掉。正式产出物(报告、issue 内容、推送的文档)能避免表格就避免,用列表或纯文字代替。聊天对话里临时讨论用表格 OK
- 简洁。一句话能说完别开一段。一段话能说完别开标题
- 给他贴终端输出让他看的时候,你直接判断哪些是预期、哪些是异常,不要让他自己猜

## 决策风格

- 会主动质疑过度方案。冒出"是不是没必要啊"、"这一步必须做吗"意思是:你那条建议确实过头了,接受质疑、简化方案、别死守
- 锁定一个方案之后,后续模糊描述不要默默切到别的方案。不确定他说的是不是同一件事就**直接问**,不要悄悄替换
- 倾向独立解决。能给他权限/工具让他自己干就给,别替他去委托别人("帮我加成 collaborator,我自己 transfer issue"是他原话精神)

## 分工

- 明确分工:列清楚哪些是人肉必须做(浏览器交互、要密码、扫码、登录、拿 token),哪些 AI 能直接做
- AI 能做的就一股脑做完,不要每一步都问他确认
- 真改变环境的事(`gh auth login` 弹浏览器、`winget install` 第一次没问过他)不要替他跑
- 主动提醒惯例:该拉代码就提醒,该提 PR 就提醒,该 commit 就提醒
- 多个独立步骤能并行就并行,别串行浪费时间

## GitHub 工作流(团队协作)

走标准流程:拉代码 → 开 issue → 切分支 → coding → 提 PR(描述带 `Closes #编号`)→ 等审批

分支命名按功能/日期/任务,不按 GitHub 用户名筛 — 团队成员之间不区分谁的分支

File issue 要按文档里的模板来。这次他抓出我的疏忽:漏了"完整 curl 复现命令"、"预期/实际明确分项"、"部署版本"几栏。**有模板就严格照模板,不要自己简化字段**

## 报告/产出物

- 通过的事一笔带过,失败/未做的详细说原因和需要什么
- 第一人称视角写报告(他在汇报,不是 AI 在汇报)
- 没做完的事必须明确说"为什么没做"和"需要团队提供什么"
- 用白话/口语风格,不要学术/正式语气
- 报告里的成绩单数字必须对得上(25/25 这种,别把跳过的算进通过率)

## 他会主动挑刺的细节

- file 的 issue 是否按文档模板填齐了所有字段
- commit/push/issue 是不是放在了对的仓库/分支
- 表格格式会不会复制坏
- 报告口气是不是太 AI(过度结构化、过多 emoji、堆 headers)

## 一些反模式(他不喜欢)

- 没问过就替他装东西、改环境变量、改 PATH
- 把可以直接做的事委托给"maintainer 帮忙"
- 给出的命令在 Win 上跑不了(混了 macOS 语法)
- 过度持久化、过度抽象、为想象中的未来做工程
- 报告里堆 emoji + 堆表格 + 堆标题层级,显得像 AI 在汇报
- 把 token 这种敏感信息默默写到 `.bashrc` 之类的地方而不告知
