# 2026-05-16 硬件输入优先机械结构（不要触控 / 不要 IMU 软件手势）

> 源：memory `feedback_prefer_mechanical_inputs.md`（originSessionId `19e87b41...`）
> 2026-05-20 更新：合并原 `feedback_imu_knock_unreliable.md`

为硬件 / 外设设计输入方式时，按 **"机械结构 > 电容触控 > IMU 软件手势"** 的顺序推荐。

## 默认优先

机械的旋钮、滚轮、扳机、摇杆、拨杆、按键 —— 这是默认。

## 降权（少推荐）

- **电容触控**（触控环 / 触控条 / 压感 / 单指敲击）—— chichu 主观感受"不太灵敏"
- **IMU 软件手势**（敲击 / 敲背 / 翻转 / 倒扣 / 摇晃）—— chichu 经验里几乎都不灵敏，容易漏触 / 误触

## 为什么

- chichu 更喜欢有反馈、有阻尼、有咔哒声的机械手感
- 触控类他原话："真的好的话我会采用"（不是黑名单，是降权）
- IMU 敲击他原话："按我的经验，敲击交互一般都非常不灵敏" —— iPhone 「轻点背面」也常被吐槽误触和漏触
- 提"零额外硬件 / 软件代替按键"的方案前，先确认机械结构能否替代；不能的话**老实加按键**

## 怎么落地

- 列方案时机械放前面、touch / IMU 手势放后面
- 标明每种方案的得失（如触控"零件少 + 静音 + 无机械磨损"）让 chichu 自己判断
- 同等手感时**优先选机械**，不要因为"触控更现代"或"零硬件"而推荐
- 出现真正机械方案做不到的场景（如 360° 滑动 + 多指 + 极薄空间）才以触控为首选，并解释为什么这次触控胜出
- **IMU 适用边界**：姿态指向（鼠标）、握姿检测、计步这类连续信号 OK；不适合用作"开关"
