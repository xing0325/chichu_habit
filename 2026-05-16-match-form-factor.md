# 2026-05-16 提议功能时先确认外设/产品形态

> 源：memory `feedback_match_form_factor.md`（originSessionId `19e87b41...`）

提议功能或扩展时，**先确认产品的物理形态、使用姿势、佩戴/握持方式**，再据此筛点子。形态不符的点子要主动剔除。

## 为什么

chichu 在 vibe coding 外设项目里被"脚踏开关接口"激怒：

> "我这个东西是拿在手上的小东西，咋可能和脚踏有关系"

他认为这是没动脑、跑火车式发散，**浪费他筛选成本**。

## 怎么落地

- 拿到硬件 / 外设 / 可穿戴类需求，第一步在心里确认：**手持？桌面？穿戴？口袋 EDC？车载？**
- 脑暴时**只列与该形态相容的方案**
- 明显冲突的直接不写：
  - 手持 → 脚踏 / 大托架
  - 穿戴 → 桌面键盘
  - 车载 → 触屏精操作
- 不确定形态时，**先问一句再发散**，别先发散十条再被打回
